# 课程所包含的知识点信息

## 1. 课程总体边界

本课程总时长约 16 小时，分 4 天，每天约 4 小时。

课程目标不是让学生记住所有代码细节，而是让学生掌握从一个 AI 产品想法出发，逐步完成产品流程、系统架构、模块边界、Agent 设计、工具扩展和核心代码实现的方法。

编码原则：

- 每天核心编码量不超过 400 行。
- 只带写最能体现架构和设计思路的核心代码。
- 繁琐代码只讲职责、输入输出和调用边界。
- 前端不讲。
- RAGFlow、TEI、Docker 只讲能力和系统位置，不讲部署细节。

## 4. 第 2 天知识点

| 模块 | 知识点 | 需要讲清楚的问题 | 讲解重点 | 编码要求 |
| --- | --- | --- | --- | --- |
| 长任务问题 | 为什么不能同步生成报告 | LLM、搜索、网页读取耗时长怎么办 | HTTP 请求只触发任务，不等待完整结果 | 不编码 |
| 后台任务 | `asyncio.create_task` | 第一版为什么用进程内后台任务 | 简单、低成本、适合 MVP；只简单对比 Celery | 核心编码 |
| 任务状态 | 任务状态机 | 前端或用户如何知道任务进展 | `queued`、`running`、`succeeded`、`failed` | 核心编码 |
| 项目状态 | 研究项目状态流转 | 项目从创建到报告完成有哪些阶段 | `brief_generating`、`outline_ready`、`outline_confirmed`、`research_running`、`report_ready` | 核心编码 |
| MongoDB | 文档数据库 | 为什么研究报告项目适合 MongoDB | 大纲、事实、洞察、报告版本都是文档型结构 | 核心编码 |
| Repository | 数据访问层 | 为什么不在 router 里直接写数据库逻辑 | 隔离数据库读写，保持接口层清晰 | 核心编码 |
| 项目持久化 | 研究项目保存 | 创建项目后系统保存什么 | topic、goal、audience、scope、status、outline | 核心编码 |
| 任务持久化 | 任务记录保存 | 后台任务如何更新状态 | task_id、project_id、task_type、status、message | 核心编码 |
| 报告持久化 | 报告版本保存 | 报告为什么要单独存版本 | report_id、html、sources、created_at | 核心编码 |
| 错误处理 | 任务失败如何记录 | Agent 或工具失败后系统如何反馈 | 不吞异常，更新任务状态和错误信息 | 核心编码 |

## 5. 第 3 天知识点

| 模块 | 知识点 | 需要讲清楚的问题 | 讲解重点 | 编码要求 |

| 工具设计 | Agent 如何扩展能力 | 为什么搜索、网页读取、RAGFlow 要做成工具 | 工具输入输出必须结构化 | 核心编码 |
| 外部搜索工具 | Tavily 或搜索 API | Agent 如何获得公开资料 | 查询词、结果标题、URL、摘要、来源 | 核心编码 |

| 结构化输出 | LLM 输出约束 | 为什么不能只拿自然语言文本 | 研究任务书、大纲、事实卡片、洞察卡片 | 核心编码 |

## 10. 课程主线提醒

讲课时始终围绕这一条线：

```text
产品想法
  -> 用户场景
  -> 业务流程
  -> 系统架构
  -> 模块边界
  -> 数据结构
  -> Agent 职责
  -> 工具扩展
  -> 核心代码
  -> 运行验证
```

不要把课程讲成框架 API 教程，也不要把课程讲成纯代码逐行实现。

学生最终需要掌握的是：以后遇到类似企业 AI 产品需求时，知道如何从零拆解、设计、实现和扩展。

当前AI很好用，那么讲项目课程，需要讲到哪些东西：
1、如何将一个具体的业务问题，转化成业务设计，例如将深度研究这个场景，怎么在业务层面进行拆分
2、如何基于这个业务拆分，通过编码实现，或者换句话说，转换成编码设计，例如，基于业务设计，设计系统调用时序图和一些关键的数据流图
3、如何基于系统调用时序图和数据流图，来设计接口出入参
4、针对于他们没有见过的API/框架，需要进行介绍，作用是什么，例如asyncio.create_task()这个API，其具体作用是是什么，为什么要使用，了解了之后，可以让AI去进行编码，都没有任何关系。最担心的就是，他们不知道有哪些框架，或者哪些工具，可以使用，然后只能无脑让AI生成代码
